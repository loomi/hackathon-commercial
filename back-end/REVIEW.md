# Review instructions

This is a NestJS 11 + Prisma + PostgreSQL backend. `CLAUDE.md` describes the
repo's conventions; treat it as authoritative project context. The rules below
override default review behavior.

## What Important means here

Reserve 🔴 Important for findings that would break production behavior, leak
data, or be hard to roll back. Specifically:

- Authentication / authorization bypass: a route that should require
  `JwtAuthGuard` but doesn't, a guard that returns `true` unconditionally, a
  query that returns rows belonging to other users when it shouldn't.
- Data exposure: passwords, password hashes, JWTs, refresh tokens, or API keys
  appearing in HTTP responses, logs, or error messages.
- Incorrect logic in service methods (wrong condition, off-by-one, swapped
  arguments, race conditions).
- Prisma queries that drop tenant / ownership scoping, or migrations that are
  not backward compatible (column rename autogenerated as drop+add, non-null
  column added without default to a non-empty table).
- Direct `process.env` reads outside `src/config/` — bypasses the typed
  config and breaks the project's config contract.
- Introducing a second ORM, a second logger, or a second error/response
  envelope. The project deliberately has one of each.

Style, naming, refactoring opportunities, and "could be more idiomatic" notes
are 🟡 Nit at most. Test-coverage suggestions are Nit unless a new
authentication-sensitive code path has zero coverage.

## Cap the nits

Post at most five 🟡 Nit comments per review. If you found more, list the rest
as a count under an "Additional nits" line in the summary. If everything you
found is a Nit, open the summary with "No blocking issues."

## Do not report

- Anything the toolchain already enforces: ESLint (`npm run lint`), Prettier
  formatting, TypeScript compile errors. Don't restate `tsc` diagnostics.
- Files under `dist/`, `node_modules/`, or any `*.lock` file.
- Generated Prisma client output (`node_modules/.prisma/`,
  `node_modules/@prisma/client`).
- Missing JSDoc on internal symbols. Public-API documentation belongs in
  Swagger decorators or `docs/`, not in code comments.
- Suggestions to add `console.log` debugging, or to replace the built-in
  `Logger` with a third-party logger.
- Recommendations to introduce barrel `index.ts` files.

## Always check

- New controllers have `@ApiTags` and every handler has `@ApiOperation` plus a
  success `@Api*Response`. Protected handlers pair `@UseGuards(JwtAuthGuard)`
  with `@ApiBearerAuth()` — flag if one is present without the other.
- New request DTOs use `class-validator` decorators. Optional fields use
  `@IsOptional()`. Email fields use `@IsEmail()`, UUID params use
  `ParseUUIDPipe`. Flag DTOs lacking validators on user-supplied fields.
- Response shapes don't leak Prisma models containing sensitive fields
  (`password`, `passwordHash`, refresh tokens). Expect a dedicated `*Entity`
  with a `fromPrisma` mapper that strips them.
- Controllers stay thin. Business logic, DB access, hashing, or
  multi-step orchestration inside a controller is a finding.
- Services throw NestJS `HttpException` subclasses, not raw `Error` or custom
  `{ error: ... }` envelopes — the global `AllExceptionsFilter` is the
  contract.
- `prisma/schema.prisma` changes ship with a matching migration directory
  under `prisma/migrations/`, and the migration SQL doesn't autogenerate a
  `DROP COLUMN ... ADD COLUMN ...` pair where a rename was intended.
- New env vars are added to both `src/config/configuration.ts` (with a typed
  field on `AppConfig`) and `.env.example`. Flag if only one is updated.
- New feature modules are registered in `src/app.module.ts` `imports`. A
  module file that exists but is never imported is a finding.
- Log lines (`Logger.log/warn/error/debug`) don't include passwords, tokens,
  full JWTs, or full request bodies for auth endpoints.
- New HTTP routes have at least an e2e happy-path test under `test/`.

## Verification bar

Before flagging a behavior claim — "this leaks X", "this races with Y", "this
returns the wrong shape" — cite the specific `file:line` evidence in the
finding. Do not infer behavior from a name alone. If the claim depends on how
a Prisma query interacts with the schema, cite the relevant model in
`prisma/schema.prisma`.

## Re-review convergence

After the first review on a PR, suppress new 🟡 Nit findings unless they were
introduced by the latest push. Continue posting 🔴 Important findings on every
push. Don't repeat findings that the author has already addressed in a
previous round.

## Summary shape

Open the review body with a one-line tally in the form
`N important, M nits, K pre-existing`. If `N` is zero, lead with "No blocking
issues" before the tally.
